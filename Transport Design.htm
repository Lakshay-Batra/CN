TRANSPORT LAYER

Design issues 

  • Multipoint-to-point transport should provide similar functionalities as TCP

    • In order delivery

    • reliability

 Sources with heterogeneous resources 

  • Cannot be server centric

    • In TCP server controls what to send & when to send

    • If multiple sources, who should control?

    • Who should maintain state information?

  • Server migration

    • Need for synchronization of status between servers

    • Who will transfer the state from old server to new one?

  • Heterogeneous resources:

    • Multiple states corresponding to each source is needed

    • Multiple state information

    • Server centric not a good idea

 

  • Receiver centric design – better idea

 Receiver controls what to send & when to send & who should send  

  • Multiple states

    • TCB state: congestion window, round-trip time, timers, etc

    • Packet reordering due to multiple sources.

  • Decoupling of functionalities

    • A multipoint-to-point connection with only one source should incur the same overheads as a point-to-point one

  • Per-connection vs. per-path functionalities

    • Buffer and socket management: per connection

    • Congestion control: per path

    • Reliability: per-connection or per-path?

  • Packet scheduling

    • Packets from multiple sources should be received in-sequence to avoid head-of-line blocking

    • minimize re-sequencing delay

    • Peer heterogeneity (bandwidth/latency mismatches and fluctuations) and peer dynamics (departures and/or arrivals)

    • Scheduling granularity vs. synchronization overheads. 

  • Sender-receiver interaction of TCP is transposed 

  • REQ –DATA handshake

    • Retain the self clocking mechanism in TCP that uses the DATA –ACK handshake

    • Allow the receiver to control which data the sender should send on a per-packet granularity 

  • REQ

    • Generated based on the available buffer space at the receiver (flow control) and the congestion window size

    • The window size controls the amount of pending REQs, while the return of DATA clocks the progression of the window (congestion control)  

  • Loss is inferred through out-of-order DATA arrivals or timeouts –the receiver has complete knowledge of the state of the received buffer (reliability)

 

  • Improved robustness using two modes: cumulative mode and pull mode

 Radical Reception Control Protocol (R2CP) 

  • RCP for multi-homed mobile hosts

    • A receiver-only extension which can communicate with one or multiple RCP senders

    • Multi-state transport protocol built atop RCP

    • Multipoint-to-point communication (as well as unicast)

 (R2CP) 

  • Receiver drives the operation of congestion control and reliability for individual paths

  • Receiver is responsible for packet scheduling from multiple sources

  • Receiver controls when, which and how much to request from each source

 Other issues 

  • Peer selection and load balancing

    • Peer departures and arrivals

    • Self-interest peers and untrusting peers

    • Fairness model

    • Deployment issues

Radical Reception Control Protocol 

  • Content replication

    • Reliability vs. timeliness

    • Requesting peers later become content suppliers

    • Out-of-band loss recovery

    • Reliable delivery can be achieved without impacting the timely delivery of the data from the (primary) source
